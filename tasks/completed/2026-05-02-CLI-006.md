## Bilan
- **Ce qui s'est bien passé** : 6 sous-tâches respectées. Découpage `render` / `prod-compose` / `env` propre, fonctions pures sans I/O. Postbuild Node natif (`fs.cp`) cross-platform a marché du premier coup. CI verte premier essai (Node 37s + Go 25s). Pattern templating réutilisable tel quel pour CLI-007 et CLI-008.
- **Ce qui a posé problème** : un seul warning Biome sur `console.log` dans le script de build → switch vers `console.info` (sortie non-debug, légitime).
- **Ce qu'on a appris** :
  - Pour les fichiers non-TS (templates, fixtures), `tsc` ne les copie pas — un postbuild script Node natif suffit, pas besoin de `cpx`/`copyfiles`.
  - Garder les générateurs **purs** (string in/out, zéro I/O) facilite drastiquement les tests et l'orchestration future.
  - L'aspect "fichier `.eta` versionné séparé" est plus lisible à long terme que des constantes string TS — particulièrement utile pour les composes Docker (syntax highlight YAML).
- **Décisions prises** : aucune ADR-level. Toutes locales et réversibles.
  - `readFileSync` + postbuild Node `fs.cp` (vs embedding TS) — pattern réutilisable
  - Apps utilisateur absentes du compose (Coolify les déploie, ADR-0005)
  - Pas de génération aléatoire de secrets (placeholders `__REPLACE_ME__` explicites)
  - Pas de commande clipanion exposée (orchestration = CLI-012)
