## Bilan

### Ce qui a bien marché

- **Cadrage produit figé en amont** (5 décisions D1-D5 + Q-A) → 0 friction de design.
- Pattern **`--dry-run` universel** (skip credentials + skip API entirely) → permet développement et tests offline, validation runtime déférée à E2E-001 par design.
- **Découpage sous-tâche 2 en Phase A** (credentials + list + remove) **puis Phase B** (add seul) → reviewability préservée malgré 10 fichiers cumulés.
- **Diagnostic upstream avant code** (pattern hérité d'ANSIBLE-001b) : Claude Code a vérifié `src/projects/coolify.ts` et `cli.test.ts` AVANT de proposer le diff, alignant les choix techniques (double quotes Biome, `runFromArgs` pattern, `fetch` natif).
- **`override readonly cause` partout** (anti-régression TS4114) → leçon INSTALL-002 intégrée à toutes les nouvelles classes Error.
- **Schémas Zod asymétriques** (input strict, output permissive) → résilience aux ajouts upstream Cloudflare sans casser l'app.
- **Pattern stat-then-load pour credentials** (cohérent backups role d'ANSIBLE-001c).
- **Test sequence avec `mockResolvedValueOnce` × N** pour vérifier ordre des appels API (notamment `--force` = list + delete + create = 3 fetch calls).
- **Factor `test-helpers.ts` en Phase B** → ferme le CLI gap noté en Phase A (pas de 3e duplication `tempCreds` / `run` / `mockJson`).
- **Disambiguation par `--content` sur remove** → couvre le cas réel de plusieurs records pour le même name+type (round-robin DNS, etc.).

### Ce qui a posé problème

- **Découverte « rclone obscure pas applicable ici »** (différent contexte que ANSIBLE-001c) — mais rappel utile que les API externes ont souvent des surprises non documentées.
- **Hiccups Biome récurrents** : `noNonNullAssertion` (`matches[0]!` → destructuring), `noUnusedTemplateLiteral` (`` `\n` `` → `"\n"`), `useTemplate` (concat → template literal). Auto-fix par `pnpm lint:fix`, mais slow le développement. CLI gap : pre-commit hook `lint:fix` automatique.
- **Tests via `runFromArgs` (Cli.run)** plus verbeux que tests directs sur la classe, mais coverage end-to-end (parsing clipanion + logique métier) justifie le coût.
- **Compteur Vitest qui passe de 144 → 164** (vs 144 stable depuis ANSIBLE-001a) signal positif mais à surveiller pour ne pas accumuler de tests de basse qualité.

### Lessons learned

- **Pattern `--dry-run` universel** doit être ajouté DÈS le cadrage initial pour toute commande qui modifie de l'état externe. Pas un flag accessoire.
- Pour les **API externes avec validation côté serveur** (Cloudflare, GitHub, etc.), asymétrie Zod (input strict / output permissive) est l'heuristique correcte. À documenter dans les conventions.
- **`override readonly cause` sur classes Error custom** = règle absolue depuis TS4114 fix d'`a63ecd1`. À ajouter dans `coding-style.md`.
- **Factor de `test-helpers.ts`** vaut le coup à 3 duplications, pas avant. YAGNI bien appliqué.
- **`runFromArgs` pattern** (`cli.test.ts` existant) > tests directs sur la classe pour les tests de commandes : exerce le parsing clipanion, détecte les bugs de paths et options.
- **Disambiguation par filter additionnel** (`--content`) > erreur « ambiguous » sans alternative. UX user-friendly fait la différence.

### Décisions structurantes prises

- **Auth Bearer token uniquement** (legacy email+key déprécié par Cloudflare 2026).
- **3 types de records MVP** : A, CNAME, TXT (couvre 95 % des besoins web app).
- **Auto-discovery zone** : `--zone` > `CLOUDFLARE_ZONE_ID` > heuristique 2-labels avec **filter strict `z.name === candidate`** (défensif contre fuzzy match API).
- **Reconciliation MVP** : impérative (`add` ne overwrite jamais sans `--force`).
- **`--force` = delete-then-create** (atomique côté ARC, gère `existing.length > 1`).
- **Default comment** `"managed-by:arc"` sur les records créés (traçabilité, omit avec `--comment=""`).
- **5 classes d'erreurs typées** (Api base + Auth + RateLimit + NotFound + Validation) avec `override readonly cause`.
- **Schémas Zod asymétriques** : input strict, output permissive.
- **`--dry-run` universel** sur `add` et `remove` (skip credentials + skip API).
- **Pattern `runFromArgs`** pour les tests de commandes (cohérent `cli.test.ts`).
- **Factor `test-helpers.ts`** à 3 duplications, pas avant.

### CLI gaps notés pour futur (10 entrées capturées)

#### Spécifiques DNS-001

1. **Headers normalization** dans `CloudflareClient.request()` : actuellement assume plain object pour `init.headers` — fragile si appelant utilise `new Headers()` ou tuples.
2. **Pagination Cloudflare API** : zones et records peuvent avoir `result_info.total_pages > 1`, MVP prend tout sans suivre — limite à 50 par défaut.
3. **Retry/backoff sur 429** `RateLimitError` (actuellement throw direct).
4. **`error_chain` parsing** (Cloudflare retourne parfois une chaîne d'erreurs imbriquées — on prend juste la première).
5. **publicsuffix list** pour TLDs composés (`.co.uk`, `.com.au` — heuristique 2-labels casse).
6. **`--force` sur `remove`** quand confirmation interactive sera ajoutée.
7. **Table layout `padEnd(38/35)`** misalignment sur noms longs — switch `cli-table3`.
8. **Zod input validation `client.createDnsRecord`** si input externe (YAML/JSON).
9. **Distribution sans test code** : exclude pattern `test-helpers.ts` dans tsconfig build.
10. **`arc dns sync` command** (déclaratif, reconciliation pure) en future tâche dédiée.

## Objectif
Livrer 3 commandes CLI `arc dns add` / `arc dns list` / `arc dns remove` qui interagissent avec l'API Cloudflare v4 pour créer/lister/supprimer des records DNS (A, CNAME, TXT) dans la zone du domaine ARC. Compatible mode `--dry-run` pour audit sans token et exécution offline en CI. Token operator-managed dans `~/.arc/credentials/cloudflare.env` (cohérent `r2.env` / `local-ai.env` du backups role).

Cette tâche est la dernière brique infra avant E2E-001 — elle débloquera le déploiement de records `A wildcard` pointant l'IP publique du VPS sur le domaine de l'opérateur.
