## Bilan

### Ce qui a bien marché

- Découpage en **3 sous-tâches** (coolify simple → ai-stack complexe → validation) avec **cadrage figé en amont** (C1-C6, A1-A8, V1-V3) → 0 friction de design.
- Pattern **« diff proposé → user valide → application »** maintenu sur tout 001b.
- **Diagnostic upstream du repo `local-ai-packaged` AVANT code** (étape 1 forcée) → a révélé 4 surprises majeures (40+ vars, JWT trio, collision port, `start_services.py`).
- `ansible-lint name[template]` détecté **proactivement par Claude Code** et fixé avant que tu le voies (auto-correction sur le diff sous-tâche 2).
- `requirements.yml` en **pré-amble** → `community.general` + `community.docker` pinés range-based, prêt pour 001b et 001c.
- `become: false` vs `true` rigoureusement mappé sur 001b sous-tâche 2 (3 user-scoped pour credentials + 8 root-scoped pour install/runtime).
- Symlink créé en position 6 (avant `start_services.py` en position 7) — **ordre crucial respecté**.
- Healthcheck pattern Coolify avec `follow_redirects: none` + status `[200, 302, 401]` + `until is succeeded` → robuste contre les retries inutiles.

### Ce qui a posé problème

- **Diagnostic upstream `local-ai-packaged` a révélé 4 trous** dans le brief initial :
  - 40+ vars dans `.env.example` (vs ~10 supposées)
  - Trio JWT cryptographiquement lié (random impossible)
  - Collision Kong 8000 ↔ Coolify 8000
  - `start_services.py` orchestrateur (sparse-clone Supabase + SearXNG dance + double-compose)
- **`arc_credentials_dir` initialement défini avec `ansible_env.HOME`** → bug sous `become: true` (résout `/root` au lieu du user pilote). Fix via expression conditionnelle `'/root' if arc_user == 'root' else '/home/' + arc_user`.
- **`ansible.builtin.command` vs `community.docker.docker_compose_v2_pull`** : pull standalone non disponible avant `community.docker` 3.7, on a opté pour `command` shell-out (CLI gap noté pour bump futur).
- **Décision JWT démo accepté avec garde-fous textuels** — risque résiduel d'oubli de regen avant exposition publique (CLI gap critique noté).

### Lessons learned

- Le pattern **« Diagnostic upstream avant code »** doit être systématique sur toute tâche qui clone/déploie un repo tiers. Sinon on découvre les surprises pendant l'implémentation.
- `ansible_env.HOME` résout différemment selon become context — utiliser une expression conditionnelle stable (basée sur `arc_user`) pour les paths user-scoped sous `become: true`.
- Les **ports exposés** en `--tags=install` doivent être inventoriés et planifiés comme ressources finies (8000 = Coolify, 8001 = Kong, 80/443 = Caddy futur, etc.).
- Quand un script upstream (`start_services.py`) gère orchestration complexe, **l'invoquer via `ansible.builtin.command` est plus défensif que le mimer en Ansible**. Coût : opacité Ansible. Bénéfice : alignement upstream.
- Le pattern **stat-then-template pour secrets** (idempotence par « preserve si existe ») a un piège : suppression manuelle entre runs = regen secrets = désync DB. À documenter dans le template lui-même.
- Pour les modules `community.*` avec features récentes : **toujours vérifier la version du module attendu vs version pinned dans `requirements.yml`** AVANT le diff.

### Décisions structurantes prises

- `/opt/local-ai/.env` symlinké vers `~/.arc/credentials/local-ai.env` (cohérent **ADR-0015**).

- **Profile Docker Compose configurable** via `arc_local_ai_profile` (défaut `"cpu"`).

### CLI gaps notés pour ANSIBLE-001c et futur

1. Si `requirements.yml` bumpe en `>=3.7,<4.0` : remplacer `command: docker compose pull` par `community.docker.docker_compose_v2_pull` (cohérence idiomatique).
2. **JWT trio démo Supabase** : régénérer impérativement avant exposition publique (Caddy + DNS + Let's Encrypt). Lien doc : https://supabase.com/docs/guides/self-hosting/docker#generate-api-keys.
3. **Pinner le sparse-clone Supabase sur SHA** quand E2E-001 stable (drift master upstream).
4. **SearXNG `cap_drop` dance** gérée par `start_services.py` upstream — surveiller les changements à chaque bump du SHA pinné.
5. **Container name Ollama** dépend du profile (`ollama-cpu` / `ollama-gpu` / ...) — vérifier le mapping si bump du profile dans la sous-tâche pull models.
6. **Suppression manuelle de `local-ai.env`** → regen secrets → perte de données Postgres. Pattern « regen = nouvelle install ». Documenter dans wiki user.
7. **Refacto possible** : monter `arc_user` au play level dans `setup.yml` `vars:` (déduplication entre rôles `hardening` + `docker` + `ai-stack` — 3 rôles maintenant avec la même expression). YAGNI 001b, à acter en 001c ou en chantier d'amélioration.
